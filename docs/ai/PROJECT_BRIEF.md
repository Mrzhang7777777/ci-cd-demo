# 项目简介

## 1. 项目目标

本项目用于搭建一个最小可部署的前后端 Demo，重点不在业务复杂度，而在于完整学习以下工程能力：

- 如何组织一个最小前后端分离项目
- 如何为前后端分别构建 Docker 镜像
- 如何使用 Docker Compose 在本地和服务器编排服务
- 如何通过 Nginx 统一对外暴露入口
- 如何使用 GitHub Actions 完成持续集成与持续部署
- 如何让云服务器只负责拉镜像和启动容器

## 2. 功能边界

第一阶段只实现最小功能闭环：

- 前端提供一个简单页面
- 页面发起后端请求
- 后端提供健康检查接口
- 后端提供一个示例业务接口

本阶段不包含：

- 登录鉴权
- 数据库
- 缓存
- 消息队列
- 文件上传
- 复杂权限系统
- 多环境高复杂度配置平台

## 3. 为什么做这个项目

很多教程只演示单点知识，比如只讲 Docker、只讲 GitHub Actions，或者只讲某个框架脚手架。这个项目要解决的是“把这些环节串成一条真实可执行交付链”的问题。

学习完成后，应当能清楚回答：

- 本地开发与生产部署的差异是什么
- 为什么服务器不应该直接拉源码后构建
- 镜像构建、镜像仓库、部署服务器之间的职责如何划分
- GitHub Actions 在 CI 与 CD 中分别做什么
- 如何让 Codex 在项目中持续产生稳定、可控的增量

## 4. 成功标准

满足以下条件即可认为第一阶段成功：

1. 本地执行 `docker compose up` 后，前后端与 Nginx 全部可用
2. 浏览器访问统一入口地址时能打开前端
3. 前端可成功请求后端 `GET /api/hello`
4. `GET /health` 可用于容器健康检查或运维探活
5. GitHub Actions 可构建并推送镜像
6. 服务器只需更新镜像并重启 Compose 服务即可完成部署

## 5. 非目标

以下内容不是当前项目第一阶段的重点：

- 高可用集群
- 自动扩缩容
- 蓝绿部署与金丝雀发布
- Kubernetes 编排
- 分布式追踪
- 完整监控告警平台

## 6. 约束条件

- 服务器配置较低：2 核 2G
- 需要控制镜像体积和运行资源占用
- 文档必须适合从零学习
- 每个任务应可独立提交，避免一次性生成大堆代码
