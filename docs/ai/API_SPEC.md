# API 约定

## 1. 目标

当前阶段只定义两个最小接口，用于验证：

- 后端服务正常运行
- 前端可以成功调用后端
- Nginx 反向代理路径配置正确

## 2. 接口列表

### 2.1 GET /health

- 用途：健康检查
- 鉴权：无
- 请求参数：无

建议响应：

```json
{
  "status": "ok"
}
```

说明：

- 用于容器健康检查
- 用于部署后快速探活
- 响应应尽量稳定、轻量、无外部依赖

### 2.2 GET /api/hello

- 用途：最小业务示例接口
- 鉴权：无
- 请求参数：无

建议响应：

```json
{
  "message": "hello from backend"
}
```

说明：

- 主要用于验证前端请求链路
- 后续可扩展为返回时间、版本号、环境标识等信息

## 3. 通用约定

- 响应格式使用 JSON
- 成功响应默认使用 HTTP 200
- 路径命名统一采用小写
- 示例接口放在 `/api/` 前缀下
- 健康检查接口独立为 `/health`

## 4. 错误处理约定

当前阶段保持最小化，不定义完整错误码体系，但建议保留以下原则：

- 返回标准 HTTP 状态码
- JSON 错误结构后续统一
- 不为最小 Demo 过早设计复杂异常中间件

## 5. 后续可扩展方向

在不破坏当前最小目标的前提下，后续可增加：

- `GET /api/version`
- `GET /api/env`
- 基础请求日志
- 统一错误响应结构
