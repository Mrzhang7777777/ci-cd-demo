# 架构说明

## 1. 总体架构

本项目采用单仓库、多服务、容器化部署的最小架构。

逻辑结构如下：

1. 用户通过浏览器访问 Nginx
2. Nginx 提供前端静态资源
3. 前端通过同域路径调用后端 API
4. Nginx 将 `/api/` 和 `/health` 请求反向代理到 FastAPI 服务
5. FastAPI 返回 JSON 响应

## 2. 规划中的服务组成

### 2.1 frontend

- 技术：Vue 3 + Vite + TypeScript
- 责任：
  - 提供最小页面
  - 发起 API 请求
  - 展示后端返回结果
- 部署方式：
  - 先构建静态资源
  - 再由 Nginx 对外提供静态文件服务

### 2.2 backend

- 技术：FastAPI + uv + Pydantic
- 责任：
  - 提供 `GET /health`
  - 提供 `GET /api/hello`
  - 提供未来 API 扩展的基础结构
- 部署方式：
  - 构建独立镜像
  - 作为应用容器运行

### 2.3 nginx

- 责任：
  - 统一对外端口
  - 托管前端静态资源
  - 反向代理后端 API
- 原因：
  - 减少浏览器跨域复杂度
  - 接近真实生产入口模式
  - 有利于后续 SSL、缓存、头部控制扩展

## 4. 请求路径规划

- `/`：前端首页
- `/assets/*`：前端静态资源
- `/api/hello`：示例接口
- `/health`：健康检查接口

## 5. 本地运行形态

本地使用 Docker Compose 编排至少以下服务：

- `backend`
- `nginx`

前端有两种候选形态：

- 方案 A：开发期使用 Vite 开发服务器，部署期使用 Nginx 静态文件
- 方案 B：本地也直接通过构建产物接入 Nginx

当前倾向：

- 为了贴近部署链路，优先采用方案 B
- 若影响学习效率，可在开发早期短暂使用方案 A

## 6. 服务器部署形态

服务器保持最小职责：

- 拉取镜像
- 加载环境变量
- 执行 `docker compose up -d`
- 清理旧镜像

服务器不负责：

- 拉取源码
- 安装前端/后端依赖
- 本地构建镜像
- 运行 CI 测试

## 7. 演进方向

在最小闭环跑通后，可以逐步增加：

- 基础日志规范
- 镜像标签策略
- 简单回滚策略
- 多环境配置分层
- 基础安全头和限流配置
