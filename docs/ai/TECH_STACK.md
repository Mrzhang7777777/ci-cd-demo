# 技术栈说明

## 1. 前端

### 1.1 Vue 3

- 用途：构建最小前端页面
- 选择原因：
  - 生态成熟
  - 上手成本适中
  - 足够覆盖组件化和构建流程学习

### 1.2 Vite

- 用途：前端开发与构建工具
- 选择原因：
  - 启动快
  - 构建链路简单
  - 适合最小演示项目

### 1.3 TypeScript

- 用途：前端类型约束
- 选择原因：
  - 提前暴露接口与数据结构问题
  - 有利于前后端接口联调

## 2. 后端

### 2.1 FastAPI

- 用途：提供 HTTP API
- 选择原因：
  - 轻量
  - 自带接口文档能力
  - 非常适合最小 API 示例

### 2.2 uv

- 用途：Python 项目依赖与运行管理
- 选择原因：
  - 速度快
  - 适合现代 Python 工程化
  - 有利于容器内构建流程简化

### 2.3 Pydantic

- 用途：数据模型与响应结构约束
- 选择原因：
  - 与 FastAPI 配合自然
  - 有利于保持接口定义清晰

## 3. 部署

### 3.1 Docker

- 用途：打包运行环境和应用
- 选择原因：
  - 保证本地、CI、服务器运行环境一致
  - 便于镜像仓库分发

### 3.2 Docker Compose

- 用途：编排本地和服务器的多容器服务
- 选择原因：
  - 语义简单
  - 足以满足单机部署
  - 学习成本远低于 Kubernetes

### 3.3 Nginx

- 用途：静态文件服务与反向代理
- 选择原因：
  - 生产实践常见
  - 配置直观
  - 适合本项目的统一入口需求

## 4. CI/CD

### 4.1 GitHub Actions

- 用途：
  - 代码检查
  - 镜像构建
  - 镜像推送
  - 远程部署触发
- 选择原因：
  - 与 GitHub 仓库天然集成
  - 足够完成本项目的全流程自动化

## 5. 镜像仓库

当前文档阶段不强制指定仓库，但建议优先考虑以下候选：

- GitHub Container Registry
- Docker Hub

当前倾向：

- 优先 GitHub Container Registry

原因：

- 与 GitHub Actions 集成顺手
- 权限和仓库关系更直观
- 适合学习以 GitHub 为中心的交付链路

## 6. 约束与取舍

- 不引入数据库，降低状态管理复杂度
- 不引入 Redis，避免多服务扩张
- 不引入 Node 服务端渲染，保持部署简单
- 不引入进程管理器，容器内保持单进程模型
